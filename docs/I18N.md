# Internacionalización (i18n)

55mas usa **next-intl v3** para contenido estático (UI) y **tablas `_translations`** en Supabase para contenido dinámico (servicios, países, etc.).

## Idiomas soportados

| Código | Idioma | Default |
|--------|--------|---------|
| `es` | Español | ✓ |
| `en` | English | |
| `pt` | Português | |
| `fr` | Français | |
| `ca` | Català | |

Configurado en `src/lib/i18n/config.ts`.

## Archivos de configuración

| Archivo | Propósito |
|---------|-----------|
| `src/lib/i18n/config.ts` | Array `locales`, `defaultLocale`, tipo `Locale` |
| `src/lib/i18n/routing.ts` | `defineRouting()` para middleware y navegación |
| `src/lib/i18n/navigation.ts` | Exports type-safe: `Link`, `redirect`, `usePathname`, `useRouter` |
| `src/lib/i18n/request.ts` | `getRequestConfig` — carga mensajes según locale |
| `src/lib/i18n/messages/{locale}.json` | Mensajes por idioma |

## Archivos de mensajes

En `src/lib/i18n/messages/`. Estructura actual:

```json
{
  "Metadata": { "title": "...", "description": "..." },
  "Common": { "loading": "...", "error": "...", "back": "...", "logout": "...", ... },
  "Placeholder": { "comingSoon": "...", "publicArea": "...", "adminArea": "...", ... }
}
```

**Convención de claves:** PascalCase para namespaces, camelCase para claves. Ejemplo: `Common.unexpectedError`.

## Uso en componentes

### Server Components (páginas, layouts)

```tsx
import { getTranslations, unstable_setRequestLocale } from 'next-intl/server';

export default async function Page({ params: { locale } }: Props) {
  unstable_setRequestLocale(locale);  // OBLIGATORIO para static generation
  const t = await getTranslations('Common');
  return <h1>{t('loading')}</h1>;
}
```

### Client Components

```tsx
'use client';
import { useTranslations } from 'next-intl';

export default function MyComponent() {
  const t = useTranslations('Common');
  return <button>{t('retry')}</button>;
}
```

Requiere estar dentro de `NextIntlClientProvider` (ya incluido en `[locale]/layout.tsx`).

### Layouts

Todo layout bajo `[locale]` debe llamar a `unstable_setRequestLocale(locale)` y exportar `generateStaticParams`:

```tsx
export function generateStaticParams() {
  return routing.locales.map((locale) => ({ locale }));
}
```

## Navegación type-safe

Importar desde `@/lib/i18n/navigation` en vez de `next/link` o `next/navigation`:

```tsx
import { Link, redirect, usePathname, useRouter } from '@/lib/i18n/navigation';
```

Estos wrappers mantienen el locale automáticamente.

## Middleware

`src/middleware.ts` encadena:
1. Supabase auth refresh (cookies)
2. next-intl locale detection y routing

Comportamiento: `/` redirige a `/es` (default). `/xx` (locale inválido) → not-found.

## Agregar un nuevo idioma

1. Crear `src/lib/i18n/messages/{code}.json` con todas las claves traducidas
2. Agregar el código al array `locales` en `src/lib/i18n/config.ts`
3. Verificar que `pnpm build` genera las páginas estáticas para el nuevo locale

## Contenido dinámico (DB)

El contenido que viene de la base de datos (servicios, ciudades, etc.) NO usa next-intl. Usa tablas `_translations` con FK a `languages(code)`:

- `service_translations(service_id, locale, name, description, includes)`
- `country_translations(country_id, locale, name)`
- etc.

**Fallback chain al leer de DB:** locale solicitado → locale default del país → `'es'`

Ver `docs/features/00-database-schema.md` para el esquema completo.
