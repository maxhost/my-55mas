# i18n strategy

55mas maneja 3 tipos diferentes de "traducción" que se almacenan en lugares distintos:

## 1. UI estática (next-intl JSON)

Strings de la UI: labels de admin, botones, mensajes de error, tabs.

- **Donde:** `src/lib/i18n/messages/{es,en,pt,fr,ca}.json`.
- **Locales soportados:** `es` (default), `en`, `pt`, `fr`, `ca`.
- **Versionado:** git. Type-safe via next-intl.

## 2. Contenido generado por usuario (NO se traduce)

Datos que el talent o cliente escribe libremente: bio, experiencia, nombre de empresa, notas. Se guardan tal como el usuario los escribió, en su idioma. No hay tabla `_translations` ni columna `i18n` para este tipo de contenido.

## 3. Catálogos definidos por admin (i18n jsonb)

Catálogos que el admin gestiona y se muestran a usuarios en su locale activo: nombres de servicios, países, ciudades, subtypes, tags, idiomas hablados, preguntas de encuesta, labels de los formularios estáticos.

### Patrón de almacenamiento

Cada entidad madre tiene una columna `i18n jsonb NOT NULL DEFAULT '{}'`:

```
countries.i18n              { "es": { "name": "España" }, "en": { "name": "Spain" }, ... }
cities.i18n                 { "es": { "name": "Madrid" }, ... }
services.i18n               { "es": { "name", "description", "includes", "hero_title", "hero_subtitle", "benefits", "guarantees", "faqs" }, ... }
service_subtype_groups.i18n { "es": { "name": "..." }, ... }
service_subtypes.i18n       { "es": { "name": "..." }, ... }
talent_tags.i18n            { "es": { "name": "..." }, ... }
spoken_languages.i18n       { "es": { "name": "..." }, ... }
survey_questions.i18n       { "es": { "label", "description?", "option_labels?": {...} }, ... }
form_definitions.i18n       { "es": { "title", "description?", "submitLabel?", "fields": { "<key>": { "label", "placeholder?", "help?" } } }, ... }
```

`services` mantiene arrays anidados (benefits/guarantees/faqs) por locale; `survey_questions.i18n[locale].option_labels` es un map adicional `{ option_value: localized_label }`.

### Fallback chain al leer

Locale solicitado → `'es'` (locale por defecto del proyecto) → `null` (o el slug/key como fallback de UI).

Implementado en `src/shared/lib/i18n/localize.ts`:

```ts
import { localize, localizedField, normalizeForMatch } from '@/shared/lib/i18n/localize';

const country = localize(row.i18n, locale);          // → { name: '...' } | null
const name = localizedField(row.i18n, locale, 'name'); // → '...' | null
```

`normalizeForMatch` baja a minúsculas, quita acentos y hace trim — útil para matchear nombres de CSV contra catálogos sin depender de case/accents.

### Escribir

Las admin save actions toman `translations: Record<locale, T>` desde la UI y serializan a `i18n jsonb` con un cast `as unknown as Json`. Para servicios (que tienen muchos locales potenciales) se usa lectura previa + merge para preservar locales no editados; para los demás se reemplaza el `i18n` entero porque el editor lo envía completo.

## Ejemplo de read típico

```ts
// Catálogo de países para un dropdown
const { data } = await supabase
  .from('countries')
  .select('id, code, i18n')
  .eq('is_active', true);

const options = (data ?? [])
  .map(c => ({
    id: c.id,
    code: c.code,
    name: localizedField(c.i18n, locale, 'name') ?? c.code,
  }))
  .sort((a, b) => a.name.localeCompare(b.name));
```

## Lo que ya NO existe

El 2026-05-02 las tablas `*_translations` separadas se consolidaron en columnas `i18n jsonb` sobre la entidad madre. Si un viejo doc menciona `country_translations`, `city_translations`, `service_translations`, `service_subtype_translations`, `service_subtype_group_translations`, `talent_tag_translations`, `spoken_language_translations`, `survey_question_translations` o `talent_profile_translations` está desactualizado.

Las views `*_localized` (`countries_localized`, `cities_localized`, `services_localized`, `categories_localized`) también desaparecieron — se reemplazaron por `localize()` en app layer.
